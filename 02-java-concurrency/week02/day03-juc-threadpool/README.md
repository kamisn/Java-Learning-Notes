
# 4.8 周三 并发笔记（可复习版）
## volatile 的作用与局限

### 1. volatile 的作用
- **保证可见性**
- **保证部分有序性**
- **不保证原子性**
> 对同一个 `volatile` 变量的写，**happens-before** 后续对这个变量的读。  
> 因此，一个线程对 `volatile` 变量的写入，对另一个线程后续的读取一定可见。

---

### 2. 为什么需要 volatile
多线程环境下，共享变量可能被缓存在 CPU 缓存或寄存器中，或被 JIT 优化，导致一个线程看不到其他线程的修改。
`volatile` 的作用：
> 让变量的读写不能被随意缓存、复用或重排序，保证一个线程写入后，其他线程后续读取时能看到最新值。
--
### 3. volatile 的局限
`volatile` **不能保证原子性**，因此无法处理并发冲突。
```java
volatile int i = 0;
i++;
```
`i++` 分为三步：读 → 计算 → 写回  
多线程下可能发生丢失更新，例如：

- 线程 A 读到 `0`
- 线程 B 也读到 `0`
- A 写回 `1`
- B 写回 `1`  
  最终结果仍为 `1`，而不是预期的 `2`。

---
## happens-before 与可见性

### 1. happens-before 的含义
> 如果 A happens-before B，那么 A 的执行结果对 B 一定可见，并且 A 的执行顺序先于 B。
- 不是代码写的先后顺序
- 是 **JMM 规定的可见性和顺序保证**
---
### 2. 常见规则（当前阶段记这四条）
- **程序次序规则**：同一线程内，前面的操作 happens-before 后面的操作
- **volatile 变量规则**：对 `volatile` 变量的写 happens-before 后续对它的读
- **监视器锁规则**：对锁的 unlock happens-before 后续对同一把锁的 lock
- **传递性**：A happens-before B，B happens-before C ⇒ A happens-before C
---
## CAS 为什么能解决 i++ 问题

### 1. CAS 的核心思想 
CAS = **Compare And Set / Compare And Swap**
逻辑：
1. 读取当前值
2. 计算新值
3. 在写回时做一次**原子比较交换**：
    - 如果内存中的值仍然等于刚才读到的旧值，就写入新值
    - 否则放弃写入，重新读取再尝试
---

### 2. CAS 解决问题的关键
> **比较旧值是否匹配** 和 **写入新值** 这两个动作由底层原子操作一次完成。
因此能避免丢失更新：
- A 和 B 同时读到旧值
- A 先成功写入
- B 写入时发现值已变，失败并重试
- B 重新读取新值，再计算，再更新
---
### 3. 哪部分是原子的
- **读取 + 计算**：不是原子的
- **CAS 的比较并写入**：是原子的
- **失败后的重试循环**：不是原子的
> 原子的是“单次更新尝试”，而不是整个自旋循环。
---
## incrementAndGet 的作用
`incrementAndGet()` 的作用：
> 对一个共享整数执行 **原子加一**，并返回加一后的结果。
常用于：计数器、序号生成器、单变量统计。
逻辑模型：
```java
do {
    prev = get();        // 读取当前最新可见值（volatile 读语义）
    next = prev + 1;     // 计算新值
} while (!compareAndSet(prev, next)); // CAS失败则重试
return next;
```
注意：
- `get()` 具有 **volatile 读语义**，读到的是当前可见的最新值
- CAS 失败后**重新读取最新值**，重新计算，重新尝试

# 一句话总结版
| 概念 | 一句话总结 |
|------|-------------|
| **volatile** | 保证可见性和部分有序性，不保证原子性 |
| **happens-before** | 规定两个操作之间什么时候存在可见性和顺序保证 |
| **i++ 为什么不安全** | 是读-改-写三步复合操作，多线程下会发生丢失更新 |
| **CAS 为什么能解决** | 把“比较旧值”和“写入新值”做成一次原子更新尝试，失败就重试 |
| **incrementAndGet** | 对单个共享变量做原子加一并返回新值 |
