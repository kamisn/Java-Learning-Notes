## 二、ConcurrentHashMap 的 put 并发控制（CAS + synchronized）

ConcurrentHashMap 通过 CAS + synchronized 的组合实现细粒度的并发控制：

### 目标桶为空的场景

多线程同时 put，且目标桶还没有数据时：

- 每个线程都用 CAS 尝试将新 Node 放入该桶

- CAS 的期望值是 `null`，要写入的新值是新的 Node 对象

- 最终只有一个线程的 CAS 操作会成功（多个线程竞争时，必然只有一个能成功）

- CAS 失败的线程：

    - 不会阻塞，而是进入自旋重试

    - 重试时会发现桶已经不为 null（被成功的线程占用了），于是进入下面的悲观锁逻辑

### 目标桶不为空的场景

当桶里已经有节点（链表 / 红黑树）时：

- 使用 `synchronized` 锁住当前桶的**头节点**

- 在锁内部执行具体的 put 逻辑：

    1. 遍历链表 / 树，找到相同 key 则覆盖 value

    2. 未找到则在链表尾部追加新节点，或是插入红黑树

    3. 必要时（满足转树条件）将链表转为红黑树

### 为什么只锁头节点？

锁住头节点就可以保证整个桶的线程安全，完全不影响其他桶的操作，因此其他桶的读写可以完全并行，大大提升了并发度。

> 🔧 细节纠正：
> 你说 “重新填入的时候就会检查到期望值已经不是 null 了，接着就用悲观锁” – 这个流程是对的，更准确的描述是：CAS 失败后会进入下一轮循环，重新判断桶的状态，若此时桶不为空，则进入 synchronized 代码块。
> 注意：不是 “把数组的位置换成链表或红黑树”，数组的引用仍然指向头节点，只是头节点后面串了更多节点，或是变为了 TreeBin 节点。

## 三、ConcurrentHashMap 的扩容机制（多线程协作）

### 扩容触发条件

满足以下任意一个条件就会触发扩容：

1. 元素总数量超过了扩容阈值 `sizeCtl`（阈值 = 容量 * 0.75，是动态变化的，不是固定值）

   >     ⚠️ 你之前写的 “一般是 64” 不对：64 是数组长度的一个阈值，不是元素数量阈值。元素数量阈值会随容量变化，比如初始容量 16 时，阈值是 12。
>
>

2. 链表长度 > 8 但数组长度 < 64（这种情况优先扩容，不转红黑树）

### 多线程协助扩容的流程

ConcurrentHashMap 支持多线程一起协助扩容，加快迁移速度：

1. **标记扩容状态**
   触发扩容的线程会修改 `volatile` 变量 `sizeCtl`，将其设置为负数，其中：

    - 高 16 位：存储扩容戳，用来标识本次扩容的目标容量

    - 低 16 位：存储当前参与扩容的线程数量

2. **任务分段**
   将旧数组分成多个任务段，每个段称为一个 “stride”，最小步长为 16（实际计算逻辑：`stride = (n >> 3) / NCPU`，如果计算结果小于 16，则取 16）。

3. **线程认领任务**
   所有线程（包括触发扩容的线程，以及其他在 put 时发现正在扩容的线程），都可以通过 CAS 从 `transferIndex` 变量中，认领一段还未处理的桶区间，处理顺序是从后往前。

4. **迁移数据**
   每个线程处理自己认领的段，对每个桶加 `synchronized` 锁，然后把桶内的节点迁移到新数组（新数组长度是旧数组的 2 倍）。
   迁移完成后，原桶的位置会放一个 `ForwardingNode` 占位符，用来标识 “该桶已经迁移完成，请到新数组查找数据”。

5. **动态任务再分配**
   如果当前线程处理完自己认领的段后，还有剩余的未处理段，会继续认领下一个段继续处理（也就是你说的 “谁先搬完谁继续搬”）。

6. **收尾工作**
   最后一个退出扩容的线程，会检查旧数组是否还有遗漏的任务，然后做最终清理：把 `nextTable` 置空，更新 `sizeCtl` 为新数组的扩容阈值。

> 你总结的 “分段长度最小为 16”、“谁先搬完谁继续搬”、“留下一个线程检查是否搬干净” 都基本正确，细节上的小差异是：实际是最后一个完成任务的线程做收尾检查，不是专门留一个线程。
>

## 四、HashSet 核心原理

HashSet 是基于 HashMap 实现的，本质就是直接复用了 HashMap 的能力：

- 存储：只关心 key，value 用一个固定的静态常量 `PRESENT`（一个空的 Object 对象）来占位

- 特点：元素不能重复，本质就是复用了 HashMap 的 key 不能重复的特性

- 添加方法：`add(e)` 底层实际调用的是 `map.put(e, PRESENT)`

    - 如果返回 null，说明添加成功，对外封装返回 true

    - 如果 key 已存在，put 会返回旧值，对外封装返回 false

> ✅ 一句话总结：HashSet 就是 HashMap 的 “key 专用版”，完全忽略了 value 部分。

## 五、LinkedHashMap 实现原理

LinkedHashMap 继承自 HashMap，在此基础上新增了顺序维护的能力：

### 整体结构

比 HashMap 多了一个独立的双向链表，用来维护元素的顺序。
节点类型是继承自 `HashMap.Node` 的 `Entry`，新增了 `before` 和 `after` 两个指针，用来维护双向链表。

### 插入顺序模式（默认模式，`accessOrder=false`）

默认情况下，LinkedHashMap 维护元素的插入顺序：
新节点插入时，会追加到双向链表的尾部，具体逻辑：

```java

// 新节点的 before 指向旧的尾节点
newEntry.before = oldTail
// 旧尾节点的 after 指向新节点
oldTail.after = newEntry
// 新节点的 after 为 null
newEntry.after = null
// 更新 tail 引用，指向新的尾节点
tail = newEntry
```

> 关键点：
>
> - `oldTail.after` 指向 `newEntry`（不是 `newEntry.after`）
>
> - `tail` 最终指向 `newEntry`（不是指向 `newEntry.after`）
>
>

最终的效果：双向链表的顺序就是元素的插入顺序，先来的在头部，后来的在尾部。

### 访问顺序模式（`accessOrder=true`，最近访问模式）

可以通过构造方法开启这个模式：

```java

new LinkedHashMap<>(16, 0.75f, true)
```

开启后，每次调用 `get(key)`，或是 put 一个已经存在的 key 时，会把当前访问的节点移动到双向链表的尾部。
效果：链表的尾部总是最近访问过的元素，头部是最久未访问的元素。
这个特性可以用来实现 LRU 缓存，配合重写 `removeEldestEntry` 方法，就可以自动淘汰最久未访问的头部元素。

### 迭代遍历

LinkedHashMap 的迭代器，会从双向链表的 `head` 开始，沿着 `after` 指针走到 `tail`，不会经过哈希桶数组。
因此迭代的顺序永远由双向链表决定，也就是插入顺序或是访问顺序，和哈希桶的位置无关。
