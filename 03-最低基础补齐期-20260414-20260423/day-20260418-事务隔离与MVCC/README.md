## 一、事务与 ACID

### 1. 什么是事务
事务是一组作为整体执行的操作，这组操作要么全部成功提交，要么全部失败回滚，不能只完成一部分。

### 2. 为什么需要事务
数据库中存在并发操作和异常场景，如果没有事务，数据很容易出现错误。事务的作用是保证数据在并发和异常情况下仍然正确。

### 3. ACID 四大特性
- 原子性：事务中的操作要么全部成功，要么全部回滚
- 一致性：事务执行前后，数据库都必须处于满足业务规则的正确状态
- 隔离性：并发事务之间应尽量互不干扰
- 持久性：事务一旦提交，结果就应该被永久保存

### 4. ACID 的实现思路
- 原子性：主要依赖 undo log 回滚日志
- 持久性：主要依赖 redo log 重做日志
- 隔离性：主要依赖 MVCC 和锁
- 一致性：依赖原子性、隔离性、持久性以及业务约束共同保证

## 二、隔离级别与并发问题

### 1. 为什么需要隔离级别
多个事务并发执行时，如果不控制事务之间的可见性和相互影响，就会出现脏读、不可重复读、幻读等问题，所以需要隔离级别来平衡一致性和性能。

### 2. 三种并发问题
- 脏读：读到了其他事务未提交的数据
- 不可重复读：同一事务中，多次读取同一行数据，结果不一致
- 幻读：同一事务中，多次按条件查询时，符合条件的记录条数发生变化

### 3. 四种隔离级别
- Read Uncommitted：可能出现脏读、不可重复读、幻读
- Read Committed：解决脏读，但仍可能出现不可重复读和幻读
- Repeatable Read：解决脏读和不可重复读；InnoDB 中会结合 MVCC 和临键锁处理很多幻读场景
- Serializable：解决脏读、不可重复读、幻读，但并发性能最差

### 4. InnoDB 默认隔离级别
InnoDB 默认隔离级别是 Repeatable Read。

## 三、MVCC

### 1. 什么是 MVCC
MVCC 是多版本并发控制。它通过保存数据的多个历史版本，让事务读取对自己可见的版本，而不是总读最新版本。

### 2. MVCC 的作用
- 减少普通读和写之间的阻塞
- 提高并发性能
- 帮助 RC、RR 实现一致性读

### 3. 快照读与当前读
- 快照读：读取对当前事务可见的历史版本，通常不加锁，普通 `select` 常见
- 当前读：读取最新版本，并通常要加锁，如 `select ... for update`、`update`、`delete`

### 4. MVCC 的核心实现
- 隐藏字段
- undo log
- 版本链
- Read View

### 5. Read View
Read View 可以粗略理解为当前事务读取数据时的一套可见性规则，它决定当前事务能看到哪个版本的数据。

### 6. RC 和 RR 的区别
- RC：每次一致性读通常生成新的 Read View，所以可能出现不可重复读
- RR：事务内通常复用同一个 Read View，因此更容易保证可重复读

## 四、锁机制

### 1. 记录锁
记录锁锁住的是已经存在的索引记录，用来防止其他事务并发修改该记录，或者对它做冲突性的当前读。

### 2. 间隙锁
间隙锁锁住的是索引记录之间的区间，主要作用是防止其他事务在这个区间内插入新记录。

### 3. 临键锁
临键锁是记录锁和间隙锁的组合，可以粗略理解为 `(前一个索引值, 当前索引值]`。

### 4. 为什么只靠记录锁不够
因为记录锁只能锁住已经存在的记录，不能阻止其他事务在范围内插入新记录，而幻读的核心是范围内记录数量发生变化。

### 5. 为什么这些锁很多时候本质上是锁索引
因为 InnoDB 依赖索引来定位记录和范围。命中索引时加锁更精确；如果没有合适索引，可能导致锁范围扩大、冲突增多、并发性能下降。

## 五、面试高频串讲

### 1. 事务和 ACID
事务是一组整体执行的操作，要么全部成功，要么全部回滚。ACID 分别是原子性、一致性、隔离性、持久性，其中原子性通常依赖 undo log，持久性通常依赖 redo log，隔离性依赖 MVCC 和锁机制实现。

### 2. 隔离级别和并发问题
隔离级别是为了解决多个事务并发时的脏读、不可重复读、幻读问题。InnoDB 默认隔离级别是可重复读。

### 3. MVCC 和锁的关系
MVCC 主要用于快照读，目的是提高普通读和写之间的并发效率；锁主要用于当前读和写操作，用来处理并发修改和范围插入问题。

### 4. 三种锁的区别
记录锁锁记录，间隙锁锁区间防插入，临键锁同时锁记录和前面的间隙，常用于处理幻读问题。
