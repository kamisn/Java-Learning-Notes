## 一、知识点梳理

### 1. 索引的本质
- 索引的本质是：**帮助数据库快速定位数据的数据结构**
- 在 InnoDB 中，索引通常使用 **B+ 树** 实现
- 索引的核心作用：
    - 减少扫描行数
    - 提高查询效率
    - 降低磁盘 I/O 成本

### 2. 为什么 MySQL 选择 B+ 树
- MySQL 更关心的是 **磁盘 I/O 次数**，而不是单纯的比较次数
- B+ 树的特点：
    - 是多叉平衡树，树高更低
    - 非叶子节点只存 key，不存完整数据
    - 叶子节点存放数据
    - 叶子节点之间通过链表连接
- B+ 树的优势：
    - 等值查询效率高
    - 范围查询效率高
    - 排序效率高

### 3. 为什么不用红黑树
- 红黑树是二叉树，每个节点能存的 key 很少，树高更高
- 数据库查询时树越高，磁盘 I/O 次数越多，因此不如 B+ 树适合数据库场景

### 4. 为什么不用 Hash 作为通用索引
- Hash 适合等值查询
- Hash 不适合范围查询
- Hash 不适合排序
- 数据库中范围查询和排序都很常见，所以 B+ 树更适合作为通用索引结构

### 5. 聚簇索引
- 聚簇索引是指：**叶子节点存放完整行数据的索引**
- 在 InnoDB 中，通常由 **主键索引** 充当聚簇索引
- 表数据本身就是按聚簇索引组织存储的

### 6. 二级索引
- 二级索引是除了聚簇索引以外的其他索引
- 二级索引的叶子节点存放的是：
    - 索引列值
    - 对应的主键值
- 二级索引的叶子节点 **不存完整行数据**

### 7. 回表
- 回表指的是：先通过二级索引查到主键值，再根据主键值去聚簇索引中查完整行数据

### 8. 覆盖索引
- 覆盖索引指的是：查询所需要的列都可以直接从索引中拿到，不需要回表

### 9. 为什么覆盖索引更快
- 覆盖索引不需要再去聚簇索引查完整行
- 少走了一次树查找
- 减少了额外的查询开销

### 10. 联合索引
- 联合索引是把多个列按照一定顺序组成的一棵 B+ 树
- 例如：`(name, age, city)`
- 联合索引的列顺序非常重要

### 11. 为什么列顺序重要
- 因为联合索引底层是按列顺序组织和排序的
- 顺序决定了查询时能否命中索引
- 顺序也决定了索引能被利用到什么程度

### 12. 最左前缀原则
- 联合索引使用时，要从最左边的列开始连续匹配
- 中间不能断
- 如果中间断了，后面的列通常无法继续被充分利用

### 13. 范围查询对联合索引的影响
- 如果联合索引中的某一列使用了范围查询，例如：
    - `>`
    - `<`
    - `>=`
    - `<=`
    - `between`
    - 部分 `like 'xx%'`
- 那么这一列后面的列通常无法继续被充分利用

### 14. 常见索引失效场景
- 不符合最左前缀原则
- 在索引列上使用函数或运算
- 左模糊查询：`like '%xx'`
- 隐式类型转换
- `or` 使用不当
- 优化器判断全表扫描更划算

### 15. 算法题：56. 合并区间
- 题目要求：把所有重叠区间合并
- 核心思路：
    - 先排序
    - 再一边扫描一边维护结果集

### 16. 为什么先排序
- 排序后区间会按左端点从小到大排列
- 当前区间只需要和结果集最后一个区间比较
- 不需要回头反复比较

### 17. 如何判断是否重叠
- 判断条件是：

```java
curr[0] <= last[1]
```

- 如果当前区间左端点小于等于结果集最后一个区间右端点，说明两个区间重叠

### 18. 如何合并区间
- 如果重叠，就更新右边界：

```java
last[1] = Math.max(last[1], curr[1]);
```

- 因为合并后的新区间必须覆盖原来两个区间的全部范围

### 19. 结果集的含义
- 结果集里始终维护的是：**已经合并好的区间**
- 每次只需要比较当前区间和结果集最后一个区间

### 20. 时间复杂度
- 排序：`O(n log n)`
- 遍历：`O(n)`
- 总复杂度：`O(n log n)`

## 二、问答易错易考点

### Q1：索引的本质是什么？
**答：**
索引的本质是帮助数据库快速定位数据的数据结构。在 InnoDB 中通常使用 B+ 树实现。

**易错点：**
不要说“索引的本质就是 B+ 树”。  
更准确的说法是：**索引本质是数据结构，InnoDB 通常用 B+ 树实现。**

---

### Q2：为什么 MySQL 不用红黑树作为主流索引结构？
**答：**
因为红黑树是二叉树，每个节点能存的 key 很少，树高更高。数据库更关注磁盘 I/O，树越高，访问磁盘页次数越多，因此不如 B+ 树适合数据库索引。

**易错点：**
不要只说“红黑树效率低”。  
更准确的说法是：**红黑树也有对数级查找效率，但数据库场景更关心磁盘 I/O，B+ 树树更矮。**

---

### Q3：为什么 B+ 树适合范围查询？
**答：**
因为 B+ 树的叶子节点之间通过链表连接，找到范围起点后，可以顺着叶子节点继续扫描。

**易错点：**
这题的核心不是“树高低”，而是：**叶子节点之间有链表连接。**

---

### Q4：什么是聚簇索引？
**答：**
聚簇索引是叶子节点存放完整行数据的索引，在 InnoDB 中通常由主键索引充当。

**易错点：**
不要说“给主键建了索引，所以顺便找到整行数据”。  
更本质的说法是：**聚簇索引的叶子节点本来就存整行数据。**

---

### Q5：二级索引的叶子节点存的是什么？
**答：**
二级索引的叶子节点存放的是索引列值和对应的主键值，而不是完整行数据。

**易错点：**
不要只说“二级索引是另一棵 B+ 树”。  
题目问的是叶子节点存什么，必须答出：**索引列值 + 主键值。**

---

### Q6：什么叫回表？
**答：**
先通过二级索引查到主键值，再根据主键值去聚簇索引中查完整行数据，这个过程叫回表。

**例子：**

```sql
select * from user where name = 'Tom';
```

执行过程：
1. 先查 `name` 的二级索引
2. 拿到对应主键
3. 再去聚簇索引中查整行数据

**易错点：**
回表不是“又查一次二级索引”，而是：**再去查主键索引，也就是聚簇索引。**

---

### Q7：什么叫覆盖索引？为什么更快？
**答：**
覆盖索引是指查询需要的列都可以直接从索引中获取，不需要回表。因为少走了一次树查找，所以通常更快。

**易错点：**
不要说“覆盖查询”，标准术语是：**覆盖索引**。

---

### Q8：什么是联合索引？为什么列顺序很重要？
**答：**
联合索引是把多个列按照一定顺序组成的一棵 B+ 树。列顺序很重要，因为它决定了索引的组织方式，也决定了查询能否命中索引以及命中到什么程度。

**易错点：**
不要只说“顺序不同，使用场景不同”。  
最好补一句：**因为底层是按列顺序排序的。**

---

### Q9：什么是最左前缀原则？
**答：**
联合索引使用时，查询条件要从最左边的列开始连续匹配，中间不能断。

**易错点：**
不要说“SQL 里写在最前面的列就行”。  
重点不是书写顺序，而是：**索引能否从最左列开始利用。**

---

### Q10：为什么 `where name='Tom' and city='Beijing'` 往往不能把 `(name, age, city)` 用满？
**答：**
因为中间缺了 `age`，不满足从左到右连续匹配，所以索引通常只能利用到 `name`，后面的 `city` 很难继续按索引顺序精确匹配。

**易错点：**
不要只回答“中间断了”。  
最好补充后果：**通常只能利用到前面的列。**

---

### Q11：为什么范围查询会影响后续列使用？
**答：**
因为范围条件会把这一列从“确定值”变成“区间”，从这一列开始，后续列很难继续保持精确有序匹配。

**易错点：**
不要把“范围查询”直接说成“模糊查询”。  
模糊查询只是范围查询的一部分场景。

---

### Q12：常见索引失效场景有哪些？
**答：**
常见情况有：
1. 不符合最左前缀原则
2. 索引列上做函数或运算
3. 左模糊查询
4. 隐式类型转换
5. `or` 使用不当
6. 优化器觉得全表扫描更划算

**易错点：**
不要说“有索引就一定会走索引”。  
正确理解是：**有索引不等于一定使用索引。**

---

### Q13：合并区间为什么一定要先排序？
**答：**
因为排序后区间会按左端点从小到大排列，遍历时当前区间只需要和结果集最后一个区间比较，不需要回头反复比较，逻辑才成立。

**易错点：**
不要只说“为了方便比较”。  
最好说完整：**排序后，当前区间只需要和结果集最后一个区间比较。**

---

### Q14：怎么判断两个区间重叠？
**答：**
判断条件是：

```java
curr[0] <= last[1]
```

因为排序后 `curr[0] >= last[0]`，只要当前区间左端点进入前一个区间右边界范围，就说明重叠。

**易错点：**
不要去比较 `curr[1]` 和 `last[1]`。  
核心是比较：**当前区间左端点和前一个区间右端点。**

---

### Q15：为什么合并时右边界要取最大值？
**答：**
因为合并后的新区间要覆盖原来两个区间的全部范围。左边界保留前一个区间左端点，右边界要取两个区间右端点的较大值。

**例子：**

```text
[1,3] 和 [2,6] => [1,6]
[1,10] 和 [2,6] => [1,10]
```

**易错点：**
不能直接写：

```java
last[1] = curr[1];
```

必须写：

```java
last[1] = Math.max(last[1], curr[1]);
```

---

### Q16：结果集 `result` 存的是什么？
**答：**
`result` 中存的是已经合并好的区间。遍历时只需要比较当前区间和 `result` 最后一个区间。

**易错点：**
不要把 `result` 理解成原数组的拷贝。  
它保存的是：**处理后的结果。**

---

### Q17：这题的贪心体现在哪里？
**答：**
这题的贪心体现在：如果当前区间能和结果集最后一个区间合并，就立刻合并；如果不能合并，就直接加入结果集。

**易错点：**
这题不是先会贪心模板才能做。  
更准确地说：**这是排序 + 扫描 + 维护结果集。**

---

### Q18：这题的时间复杂度是多少？
**答：**
时间复杂度是 `O(n log n)`，因为主要耗时在排序，后面的遍历合并是 `O(n)`。

**易错点：**
不要只说“遍历是 O(n)”。  
这题整体复杂度要把排序算进去。

---

### Q19：这题代码里最容易写错的地方有哪些？
**答：**
常见错误：
1. `ArrayList` 拼写错误
2. `return` 写进 `for` 循环里
3. 重叠后右边界没有取最大值
4. 忘记处理空数组
5. 最后忘记把 `List<int[]>` 转成 `int[][]`

## 三、代码模板

### 1. 合并区间正确代码

```java
import java.util.*;

class Solution {
    public int[][] merge(int[][] intervals) {
        if (intervals == null || intervals.length == 0) {
            return new int[0][];
        }

        Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));

        List<int[]> result = new ArrayList<>();
        result.add(intervals[0]);

        for (int i = 1; i < intervals.length; i++) {
            int[] last = result.get(result.size() - 1);
            int[] curr = intervals[i];

            if (curr[0] <= last[1]) {
                last[1] = Math.max(last[1], curr[1]);
            } else {
                result.add(curr);
            }
        }

        return result.toArray(new int[result.size()][]);
    }
}
```

## 四、今日易错总结

### MySQL 易错点
- 不能说“索引本质就是 B+ 树”，更准确是索引本质是数据结构，InnoDB 通常用 B+ 树实现
- 聚簇索引的关键不是“主键能找到整行”，而是叶子节点本来就存整行数据
- 二级索引叶子节点存的是索引列值 + 主键值
- 覆盖索引不要说成覆盖查询
- 有索引不代表一定会走索引

### 算法易错点
- 合并区间前必须先排序
- 判断是否重叠看的是 `curr[0] <= last[1]`
- 更新右边界必须用 `Math.max`
- `return` 不能写进循环里
- `ArrayList` 不要拼错
- 注意空数组处理
