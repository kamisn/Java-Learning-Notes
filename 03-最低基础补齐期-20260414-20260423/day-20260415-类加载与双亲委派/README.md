4 月 15 日学习笔记
一、知识点整理
1. 类加载过程

JVM 中，类加载过程一般分为五个阶段：

1.1 加载（Loading）
把已经编译好的 .class 字节码文件读入内存
在方法区中生成类的元信息
在堆中生成对应的 Class 对象，作为访问该类的入口
1.2 验证（Verification）
检查字节码是否符合 JVM 规范
保证类加载的安全性和正确性
1.3 准备（Preparation）
给 类变量（static 变量） 分配内存
给它们赋 默认值
这里赋的不是程序员写的值，而是默认零值

示例：

static int a = 10;

在准备阶段，a 的值先是 0。

1.4 解析（Resolution）
把常量池中的 符号引用 替换成 直接引用

可以先粗略理解为：

符号引用：用名字描述目标
直接引用：能直接定位到目标
1.5 初始化（Initialization）
执行类构造器 <clinit>
给静态变量赋程序员真正写的值
执行静态代码块

示例：

static int a = 10;

在初始化阶段，a 才真正变成 10。

1.6 核心结论

准备阶段给 static 变量赋默认值，初始化阶段才赋程序员真正写的值。

2. 准备阶段和对象创建时零值初始化的区别

这两者很像，但不是一回事。

2.1 准备阶段
属于 类加载流程
处理的是 类变量（static 变量）
给 static 变量赋默认值
2.2 对象创建时零值初始化
属于 对象创建流程
处理的是 实例变量
在对象分配内存后，先给实例字段赋默认值

示例：

class Demo {
static int a = 10;
int b = 20;
}
a：准备阶段先变 0，初始化阶段变 10
b：对象创建时先变 0，之后再按初始化逻辑变成 20
2.3 核心结论

前者处理类变量，后者处理实例变量；前者属于类加载，后者属于对象创建。

3. .class 文件、Class 对象、反射的关系
   3.1 .class 文件
   指 Java 源码编译后生成的字节码文件
   类加载阶段加载的是这个字节码文件
   3.2 Class 对象
   JVM 加载类后，会在运行时生成对应的 Class 对象
   可以通过以下方式拿到：
   User.class
   obj.getClass()
   Class.forName(...)
   3.3 和反射的关系

Spring 或 Java 反射机制经常是先拿到 Class 对象，再去获取：

构造器
方法
字段
注解
3.4 核心结论

类加载里讲的 .class 更偏字节码文件；反射里常说的 .class 更偏运行时的 Class 对象。

4. 元信息是什么

元信息可以理解为：类在运行时的一份结构说明书。

通常包括：

类名
父类名
实现的接口
字段信息
方法信息
构造器信息
访问修饰符
运行时常量池
注解等附加信息
核心结论

元信息本质上就是 JVM 保存的类结构描述。

5. static 变量和常量池的关系

二者都和类相关，但不是一回事。

5.1 static 变量
属于类变量
是类真正拥有的变量
5.2 常量池
保存常量信息和符号引用
更像类定义中的“资源表”
5.3 核心结论

常量池不是 static 变量，static 变量也不是常量池。

6. 双亲委派机制
   6.1 定义

双亲委派是指：

类加载器在接到类加载请求后，不会先自己加载，而是先把请求委托给父加载器处理；只有父加载器无法完成加载时，当前加载器才会自己尝试加载。

这里的“父”不是 Java 继承里的父类，而是 类加载器之间的层级关系。

7. 双亲委派的执行逻辑

整体流程可以理解为：

当前加载器收到类加载请求
先向上委托父加载器
父加载器继续往上委托
直到更高层加载器尝试处理
如果上层加载器能完成加载，就由上层加载
如果上层加载器无法完成，再逐层返回
最终由当前加载器加载自己职责范围内的类
核心结论

双亲委派是“父优先”，不是“子先加载再被替换”。

8. 双亲委派的作用

主要有两个：

8.1 避免类重复加载

同一个类不需要被多个加载器反复加载。

8.2 保护核心类库

比如用户自定义的 java.lang.String，不能替代 JDK 中真正的 String。

核心结论

双亲委派的两大作用：避免重复加载、保护核心类库。

9. 为什么说是“父优先”，不是“父全包”

因为父加载器也有自己的职责边界。

父能加载：父来加载
父加载不了：请求返回子加载器
子加载器再加载自己职责范围内的类
核心结论

父加载器虽然优先，但不是所有类都由它负责。

10. 为什么不准确说“父加载器覆盖了子加载器的类”

真实流程不是：

子加载器先把自己的类加载好了
再被父加载器替换掉

而是：

子加载器先把请求向上委托
父加载器如果能处理，请求直接结束
子加载器根本没有机会再定义自己的同名类
核心结论

更准确的说法是：父加载器优先处理成功，子加载器没有机会再加载同名类。

11. Application ClassLoader 的作用

Application ClassLoader 不是只负责“提交请求”。

它的真实作用是：

先按双亲委派把请求向上交
如果父加载器处理不了
再由它自己加载应用程序中的普通类

例如项目里的：

UserController
UserService

这些类通常最终就是由 Application ClassLoader 加载的。

核心结论

Application ClassLoader 不是只负责传请求，它也是普通应用类的实际加载者。

12. 类初始化的触发条件

类加载不等于类初始化。

常见会触发类初始化的情况有：

12.1 new 一个对象
new User();
12.2 访问类中非编译期常量的静态变量
System.out.println(User.a);
12.3 调用类的静态方法
User.test();
12.4 通过反射主动使用类
Class.forName("com.demo.User");
12.5 初始化子类时，父类会先初始化
12.6 程序启动时，main 方法所在的类会先初始化
核心结论

类加载不等于类初始化；只有主动使用类时，才会触发初始化。

13. 编译期常量

编译期常量指的是：

在编译阶段就能确定下来的常量。

示例：

static final int A = 10;
static final String S = "abc";

访问这种常量时，编译器可能直接把值内联到调用方字节码中，因此通常不会触发定义该常量的类初始化。

核心结论

访问编译期常量通常不会触发类初始化，因为值可能已经在编译阶段被内联。

14. Child.num 为什么可能初始化父类

示例：

class Parent {
static int num = 10;
}

class Child extends Parent {
}

访问：

System.out.println(Child.num);

触发初始化的通常是 父类 Parent，不是子类。

原因是：

num 真正定义在父类中
主动使用的是父类的静态成员
子类只是借用了访问形式
核心结论

访问的是谁定义的静态成员，通常初始化的就是谁。

15. 静态变量是不是对象

静态变量本身 不等于对象。

它只是类的成员变量，是否和对象有关，要看类型。

15.1 不是对象的例子
static int a = 10;
15.2 引用类型的例子
static User user = new User();

这里：

user 是静态变量
它存的是对象引用
真正的对象是 new User() 创建出来的实例
核心结论

静态变量是不是对象，要看它的类型；变量本身不等于对象本体。

16. 访问静态变量为什么可能触发初始化，但不等于创建对象

访问静态变量时，JVM 需要先确保这个类已经初始化好，因为静态变量属于类。

但这不等于创建对象，因为：

没有执行 new
没有产生新的实例对象
只是主动使用了类的静态成员
核心结论

访问静态变量触发初始化，不等于在创建对象。

17. 算法题：组合和子集的区别
    17.1 子集题
    任意长度的 path 都可能是合法答案
    所以每到一个节点都可以收集当前 path
    17.2 组合题
    题目要求必须选出 恰好 k 个元素
    所以只有当 path.size() == k 时，当前路径才是合法答案
    核心结论

子集题每层都能收集，组合题只有长度等于 k 才能收集。

18. 为什么组合题中 path.size() == k 后要立刻 return

因为：

当前路径已经是合法答案
继续往下选会超过 k
不再符合题意

所以要：

先收集答案
立刻返回
回溯尝试其他分支
核心结论

组合题长度一旦达到 k，当前分支就该结束。

19. 为什么递归传 i + 1，而不是 startIndex + 1

因为下一层递归处理的是：
当前选中的这个元素后面的候选范围
而不是当前层起点的后一个位置。
正确写法：
backtracking(n, k, i + 1);
错误倾向：
backtracking(n, k, startIndex + 1);
否则可能导致重复选取，出现类似 [2,2] 的错误结果。
核心结论
下一层递归应该基于当前选中的 i 往后走，而不是机械地让 startIndex + 1。

二、面试问答整理
1. 类加载过程有哪些阶段？每个阶段做什么？
答：
JVM 中，类加载过程一般分为五个阶段：加载、验证、准备、解析、初始化。
加载阶段会把已经编译好的 .class 字节码文件读入内存，并生成对应的 Class 对象。验证阶段会检查字节码是否符合 JVM 规范，保证安全性和正确性。准备阶段会给 static 变量分配内存并赋默认值。解析阶段会把常量池中的符号引用替换成直接引用。初始化阶段会执行类构造器 <clinit>，给静态变量赋程序员真正写的值，并执行静态代码块。

2. 准备阶段和初始化阶段有什么区别？
答：
准备阶段和初始化阶段的核心区别在于赋值内容不同。准备阶段处理的是类变量，也就是 static 变量，它会给这些变量分配内存并赋默认值。初始化阶段才会执行 <clinit>，把程序员真正写的赋值逻辑执行掉，同时执行静态代码块。
例如 static int a = 10，准备阶段时 a 先是 0，初始化阶段时才变成 10。

3. 准备阶段给 static 变量赋默认值，和对象创建时实例变量先拿到零值，有什么区别？
答
它们的区别主要有两点。
第一，发生的流程不同。准备阶段属于类加载流程，而对象成员变量先拿到零值属于对象创建流程。
第二，作用对象不同。准备阶段处理的是类变量，也就是 static 变量；对象创建时赋零值处理的是实例变量，因为它属于具体对象。

比如 static int a = 10，在准备阶段 a 先是默认值 0，到初始化阶段才变成 10。而 int b = 20 是在 new 对象时，b 先拿到默认值 0，之后再按程序员写的逻辑变成 20。

4. .class 文件和反射里的 Class 对象是什么关系？
答：
类加载里说的 .class 文件，指的是 Java 源码编译之后生成的字节码文件。JVM 会把这个字节码文件加载进内存，并在运行时生成对应的 Class 对象。
反射里常见的 User.class、obj.getClass()、Class.forName() 拿到的，都是运行时的 Class 对象。之后 Spring 或 Java 反射就是基于这个 Class 对象去获取方法、字段、构造器和注解等信息。

5. 元信息是什么？
答：
元信息可以理解为类在运行时的一份结构说明书。通常包括类名、父类、接口、字段、方法、构造器、访问修饰符、运行时常量池以及注解等信息。JVM 后续创建对象、反射调用方法，都要依赖这些信息。

6. 双亲委派是什么？
答：
双亲委派是指类加载器在接到类加载请求后，不会先自己加载，而是先把请求委托给父加载器处理。这里的父不是 Java 继承中的父类，而是类加载器之间的层级关系。只有父加载器无法完成加载时，当前加载器才会自己尝试加载。

7. 双亲委派有什么作用？
答：
双亲委派的核心作用主要有两个：第一，避免同一个类被重复加载；第二，保护 JDK 核心类库，防止用户自定义同名类替代核心类，比如 java.lang.String。

8. 为什么双亲委派可以保护核心类库？
答：
因为双亲委派采用的是父加载器优先机制。当加载 java.lang.String 这种核心类时，请求会先向上委托，而上层加载器本来就负责加载 JDK 核心类库，所以最终优先加载的是 JDK 自带的类。这样用户自定义的同名类就无法替代核心类，从而保证了核心类库的安全。

9. 为什么说双亲委派是“父优先”，而不是“父全包”？
答：
因为父加载器虽然优先，但它也有自己的职责边界，并不是所有类都由它负责加载。子加载器把请求向上委托后，如果父加载器能处理，就由父加载器加载；如果父加载器处理不了，请求会再回到子加载器，由子加载器加载自己职责范围内的类。所以它是父优先，而不是父全包。

10. 为什么不准确说“父加载器覆盖了子加载器的类”？
答：
因为双亲委派的真实流程不是子加载器先加载好了自己的类，再被父加载器替换掉。实际情况是：子加载器收到类加载请求后，先把请求向上委托，如果父加载器能够处理，这个请求就直接由父加载器完成，子加载器根本没有机会再定义自己的同名类。所以更准确的说法是：父加载器优先处理成功，子加载器没有机会再加载同名类。

11. Application ClassLoader 的作用是什么？
答：
Application ClassLoader 不是只负责向上提交请求，它也是普通应用类的实际加载者。它会先按照双亲委派把请求向上交，如果父加载器加载不了，请求会再回到它这里，由它自己加载应用程序中的普通类，比如项目中的 UserController、UserService 等。

12. 什么情况下会触发类初始化？
答：
常见会触发类初始化的情况有：new 这个类的对象、访问类中非编译期常量的静态变量、调用类的静态方法、通过反射主动使用类、初始化子类时父类会先初始化，以及程序启动时 main 方法所在的类会先初始化。

13. 什么是编译期常量？为什么访问它通常不会触发类初始化？
答：
编译期常量是指在编译阶段就能确定下来的常量，通常是 static final 修饰，并且赋值是编译器能够直接计算出来的字面量或常量表达式，比如 static final int A = 10，或者 static final String S = "abc"。
访问这类常量通常不会触发类初始化，因为编译器可能在编译时就把这个值直接内联到调用方的字节码里，运行时不需要再主动使用定义这个常量的类。

14. 为什么访问 Child.num，如果 num 定义在父类里，初始化的是父类而不是子类？
答：
因为 num 这个静态变量真正定义在父类里，所以虽然代码写成 Child.num，但本质上主动使用的是父类的静态成员，因此触发初始化的是父类，而不是子类。如果这个静态变量又不是编译期常量，那么这次访问就会触发父类初始化。

15. 静态变量是不是对象？为什么？
答：
静态变量不一定是对象。它本质上只是类的成员变量，是否和对象有关，要看它的类型。如果是 static int a，那它只是一个基本类型静态变量，不是对象；如果是 static User user，那它是一个引用类型静态变量，里面保存的是对象引用，但变量本身也不等于对象本体。

16. 为什么访问静态变量也可能触发类初始化，但这不等于在创建对象？
答：
访问静态变量也可能触发类初始化，是因为静态变量属于类，JVM 在主动使用这个类的类级别成员之前，需要先确保这个类已经初始化好。但这不等于在创建对象，因为这里并没有执行 new，也没有产生新的实例对象，只是在使用类本身的静态成员。

17. 组合题和子集题的核心区别是什么？
答：
子集题中，任意长度的 path 都可以看成一个合法子集，所以每到一个节点都可以把当前 path 的快照加入结果集。
而组合题对答案长度有明确要求，只有当 path 中元素个数恰好等于 k 时，它才是一个合法组合，所以不能像子集题那样每层都收集。

18. 为什么组合题中 path.size() == k 后要立刻 return？
答：
因为当 path.size() == k 时，当前路径已经是一个合法答案了，而且组合题要求的长度就是 k，继续往下选只会让路径长度超过 k，不再符合题意。所以这时候要先收集答案，然后立刻 return，回溯去尝试其他分支。

19. 为什么递归调用传 i + 1，而不能写 startIndex + 1？
答：
因为下一层递归要处理的是“当前选中的这个数后面的候选范围”，所以应该传 i + 1，而不是 startIndex + 1。startIndex 只是这一层循环的起点，不代表当前真正选中的元素。只有用 i + 1，才能保证下一层从当前元素后面开始选，避免重复选择同一个数，比如出现 [2,2] 这样的结果。
