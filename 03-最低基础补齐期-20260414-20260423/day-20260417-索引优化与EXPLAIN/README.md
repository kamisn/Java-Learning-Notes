# 一、MySQL `EXPLAIN`
## 1. `EXPLAIN` 是什么
`EXPLAIN` 用来查看一条 SQL 的执行计划。
它的核心作用不是返回查询结果，而是告诉我们：
- MySQL 准备怎么执行这条 SQL
- 有没有使用索引
- 使用了哪条索引
- 预估扫描了多少行
- 有没有额外排序、临时表等性能问题
可以把它理解成：
> `EXPLAIN` 是分析 SQL 执行方式和排查索引问题的重要工具。
---
## 2. 为什么学了最左前缀和索引失效之后，还要学 `EXPLAIN`

因为：

- **最左前缀、索引失效** 属于 **规则层**
- **`EXPLAIN`** 属于 **实际执行层**

前者告诉你“理论上应该怎样”，后者告诉你“数据库实际上是怎么做的”。

正确流程是：

> 先根据索引规则做预测，再用 `EXPLAIN` 验证 SQL 是否真的按预期使用了索引。

---

## 3. `EXPLAIN` 重点字段

今天重点掌握这 5 个：

### 3.1 `type`
表示 MySQL 访问表的方式，可以粗略反映这条 SQL 查得好不好。

常见值的理解：

- `ALL`：全表扫描，通常不是理想情况
- `index`：全索引扫描，虽然用了索引，但不一定高效
- `range`：范围扫描
- `ref`：非唯一索引等值查询
- `eq_ref`：唯一索引或主键关联查询，通常更好
- `const`：主键或唯一索引等值查询，通常很好

当前阶段至少记住：

> `ALL` 一般差，`range/ref` 常见，`const/eq_ref` 通常更好。

---

### 3.2 `possible_keys`
表示：

> 这条 SQL 理论上可能用到哪些索引

它只是候选名单，不代表一定会真正使用。

---

### 3.3 `key`
表示：

> MySQL 最终实际使用的索引

这是真正要重点看的字段。

---

### 3.4 `rows`
表示：

> MySQL 预估要扫描的行数

一般来说：

> `rows` 越大，通常说明查询代价越高。

但不能机械理解为“`rows` 大就一定是全表扫描”，也可能是筛选条件不够精准，或者索引利用不够好。

---

### 3.5 `Extra`
表示补充信息，常见的重点值有：

#### `Using where`
表示 MySQL 还需要进一步根据条件过滤数据。

这个不一定是坏事，很多正常查询都会出现。

#### `Using index`
表示查询所需要的数据可以直接从索引中拿到，不需要回表。

很多时候属于：

> 覆盖索引

通常是好信号。

#### `Using filesort`
表示 MySQL 不能直接按索引顺序得到结果，需要额外排序。

要警惕，因为会增加 CPU 和内存开销，数据量大时性能会变差。

#### `Using temporary`
表示 MySQL 在执行过程中使用了临时表来处理中间结果。

通常出现在：

- `GROUP BY`
- `DISTINCT`
- 某些复杂排序
- 联表后再分组/排序

也要警惕，因为会增加额外开销。

---

## 4. 如何快速判断有没有走索引

当前阶段可以用这个简化判断模板：

### 先看 `key`
- 如果 `key = NULL`，通常说明没有实际使用索引
- 如果 `key` 有值，说明实际使用了某条索引

### 再看 `type`
- 如果 `type = ALL`，通常说明全表扫描
- 如果 `type = ref / range / const / eq_ref`，通常说明索引参与了查询

### 再看 `rows`
- `rows` 越大，通常查询越重

### 最后看 `Extra`
重点看是否有：
- `Using filesort`
- `Using temporary`
- `Using index`

---

## 5. 看 `EXPLAIN` 的推荐顺序

当前阶段建议按这个顺序看：

1. `key`
2. `type`
3. `rows`
4. `Extra`
5. `possible_keys`

---

## 6. `type = index` 和 `Extra = Using index` 的区别

这两个名字很像，但不是一回事。

### 6.1 `type = index`
表示：

> MySQL 对索引做了全索引扫描

它说明“用了索引”，但不代表是精准查找。

本质上更接近：

> 把整棵索引树扫了一遍

所以它虽然比 `ALL` 更可能节省一些代价，但通常不如 `ref / range / const` 理想。

---

### 6.2 `Extra = Using index`
表示：

> 查询所需要的数据可以直接从索引中拿到，不需要回表

这个通常意味着：

> 覆盖索引

所以它一般是好信号。

---

### 6.3 二者的本质区别

- `type = index`：强调 **怎么查**
- `Extra = Using index`：强调 **数据从哪里拿，需不需要回表**

它们甚至可能同时出现，但含义完全不同。

---

## 7. 结合最左前缀和索引失效理解 `EXPLAIN`

假设有联合索引：

```sql
(name, age, city)
````

---

### 7.1 只查 `age = 20`

不符合最左前缀原则。

常见判断：

* `key = NULL`
* `type = ALL`

也就是：

> 这条联合索引通常不能被有效利用。

---

### 7.2 查 `name = 'Tom' AND city = 'Beijing'`

这里跳过了中间列 `age`。

注意：

> 即使 `key` 可能显示联合索引名，也不代表 `name、age、city` 三列都被完整高效利用了。

更准确地说：

> 联合索引通常只能高效利用到 `name` 这一段，后面的 `city` 往往只能继续做过滤。

这里很容易出现：

* `key` 有值
* `Extra` 里有 `Using where`

---

### 7.3 查 `name = 'Tom' AND age > 20 AND city = 'Beijing'`

这里 `age > 20` 是范围查询。

一旦联合索引中间列进入范围查询，后面的列通常就不能继续像等值匹配那样高效参与索引定位。

所以：

* `key` 可能仍然是这个联合索引
* 但 `city` 往往无法继续高效缩小扫描范围
* `Extra` 常见会出现 `Using where`

如果 `SELECT` 的列刚好都在联合索引里，也可能同时出现 `Using index`。

---

### 7.4 对索引列做函数

例如：

```sql
YEAR(create_time) = 2024
```

如果索引建在 `create_time` 上，那么数据库保存的是原始列值的有序结构，而不是函数计算后的结果。

所以：

> 对索引列做函数，数据库往往无法直接利用原索引的有序性定位数据，容易导致索引失效或利用变差。

常见表现：

* `key = NULL`
* `type = ALL`

---

### 7.5 `LIKE '%xx'`

以 `%` 开头的模糊查询，前缀不固定，索引通常难以正常利用。

常见表现：

* `key = NULL`
* `type = ALL`

而如果是：

```sql
LIKE 'Tom%'
```

则更有机会使用索引。

---

## 8. `EXPLAIN` 这一部分的核心总结

### 8.1 规则和验证的关系

* 最左前缀、索引失效：讲规则
* `EXPLAIN`：讲实际执行结果

### 8.2 当前阶段最值钱的判断方式

拿到一条 `EXPLAIN`，先问自己四件事：

1. `key` 是不是 `NULL`
2. `type` 是不是 `ALL`
3. `rows` 大不大
4. `Extra` 里有没有 `Using filesort` 或 `Using temporary`

# （一）MySQL `EXPLAIN` 面试问答

## 1. 什么是 `EXPLAIN`？

`EXPLAIN` 用来查看 SQL 的执行计划，分析 MySQL 实际准备怎么执行一条 SQL，例如是否使用索引、扫描了多少行、是否有额外排序或临时表等。

---

## 2. 为什么学了最左前缀和索引失效之后，还要学 `EXPLAIN`？

因为最左前缀和索引失效讲的是规则，`EXPLAIN` 讲的是实际执行结果。判断 SQL 是否真的按预期使用了索引，需要通过 `EXPLAIN` 验证。

---

## 3. `possible_keys` 和 `key` 的区别是什么？

* `possible_keys`：理论上可能使用的索引
* `key`：MySQL 最终实际使用的索引

---

## 4. `rows` 表示什么？它的值大说明什么？

`rows` 表示 MySQL 预估要扫描的行数。一般来说，值越大，说明查询代价通常越高，但不一定就意味着全表扫描，也可能是筛选条件不够精准或索引利用不充分。

---

## 5. `type` 这个字段主要看什么？

`type` 表示 MySQL 访问表的方式，可以粗略判断这条 SQL 查得好不好，例如是全表扫描、范围扫描，还是索引等值查询。

---

## 6. `Using filesort` 和 `Using temporary` 分别说明什么？

* `Using filesort`：说明 MySQL 不能直接按索引顺序得到结果，需要额外排序
* `Using temporary`：说明 MySQL 在执行过程中使用了临时表来处理中间结果

这两个都要警惕，因为通常意味着额外性能开销。

---

## 7. 判断一条 SQL 有没有真正用到索引，最先看哪个字段？

最先看 `key`，因为它表示 MySQL 最终实际使用的索引。

---

## 8. 如果 `key = NULL`，通常说明什么？

通常说明这条 SQL 没有实际使用索引。

---

## 9. 如果 `type = ALL`，通常说明什么？

通常说明这条 SQL 进行了全表扫描，一般不是理想情况。

---

## 10. 如果 `key` 有值，`type = ref`，`rows` 也不大，可以大致判断什么？

可以大致判断这条 SQL 使用了索引，访问方式正常，扫描数据量也不大，整体执行情况通常还可以。

---

## 11. `Using index` 为什么一般是好信号？

因为它通常表示查询所需的数据可以直接从索引中获取，不需要回表，很多时候属于覆盖索引，因此查询代价更低。

---

## 12. `type = index` 和 `Extra = Using index` 的区别是什么？

* `type = index`：表示全索引扫描，强调访问方式
* `Extra = Using index`：表示查询所需数据可以直接从索引中拿到，不需要回表，强调取数方式

两者不是一回事。

---

## 13. 为什么说 `type = index` 虽然用了索引，但不一定理想？

因为它表示 MySQL 对索引做了全索引扫描，虽然用了索引，但不是精准定位到少量记录，通常不如 `ref / range / const` 这类访问方式高效。

---

## 14. 为什么 `Extra = Using index` 往往是好信号？

因为它通常表示查询所需列都在索引中，可以直接从索引记录返回结果，不需要回表，很多时候属于覆盖索引。

---

## 15. 联合索引是 `(name, age, city)`，如果 SQL 只查 `age = 20`，应该怎么判断？

因为跳过了最左列 `name`，不符合最左前缀原则，所以这条联合索引通常不能被有效利用。`EXPLAIN` 中常见表现是：

* `key = NULL`
* `type = ALL`

---

## 16. 如果 SQL 是 `name = 'Tom' AND city = 'Beijing'`，为什么即使 `key` 有值，也不代表条件都被完整高效利用了？

因为 `key` 只表示 MySQL 选中了哪条索引，不表示联合索引中的每一列都被完整利用。对于 `(name, age, city)` 来说，跳过了中间列 `age`，通常只能高效利用到 `name` 这一段，后面的 `city` 往往只能继续做过滤。

---

## 17. 为什么范围查询 `age > 20` 之后，后面的列 `city` 通常会受影响？

因为联合索引中一旦中间列进入范围查询，后续列通常不能再继续像等值匹配那样高效参与索引定位，所以 `city` 往往只能作为额外过滤条件。

---

## 18. 为什么对索引列做函数，容易导致索引失效？

因为索引保存的是原始列值的有序结构，而不是函数计算后的结果。对索引列做函数后，数据库往往无法直接利用原索引的有序性定位数据，因此索引容易失效或利用变差。

---

## 19. 学 `EXPLAIN` 和学“最左前缀 / 索引失效”之间，正确关系是什么？

最左前缀和索引失效讲的是规则，`EXPLAIN` 讲的是实际执行结果。正确做法是先根据规则预测，再用 `EXPLAIN` 验证 SQL 是否真的按预期使用了索引。
